LinkedTransferQueue(2)-成员变量和核心方法

LinkedTransferQueue中的大部分方法(add、put、offer、poll、take、transfer、tryTransfer)都是由xfer()方法实现的，所以xfer()方法是所有方法的核心。

内容

  1. 成员变量
  2. 重要成员方法
    2.1 xfer()：向队列插入或者取走数据对象的方法。
    2.2 tryAppend()：尝试将节点插入到队列尾部(如果与队列中的节点同类型，则可以插入)。
    2.3 awaitMatch()：对于需要线程等待(transfer()、take()等方法)的方法，调用这个方法进行等待。
    2.4 unsplice()：清理队列中匹配成功或者取消匹配的节点。

成员变量

  1. MP：表示当前os环境是否允许是多线程的环境。静态变量所有LinkedTransferQueue对象共用。
  2. FRONT_SPINS：表示队列第一个节点的阻塞前可以进行的自旋次数。默认2^7次，即128次。也是静态变量。
  3. CHAINED_SPINS：表示如果当前节点的前驱正在自旋而非阻塞，则当前节点的阻塞前可以进行的自旋次数。值为FRONT_SPINS的一半，即为64次。也是静态变量。
  4. SWEEP_THRESHOLD：即当一个节点匹配成功、remove()移除、取消(中断或者超时)调用unsplice()方法时，遭遇两种特殊情况而清理失败时，unsplice()失败的最大次数。当达到这个阀值之后，进行一次完成的清理。
  1. head：表示队列头部，初始为null，直到第一个节点插入。
  2. tail：表示队列尾部，初始为null。
  3. sweepVotes：表示unsplice失败次数计数，当到达SWEEP_THRESHOLD时，进行全局sweep()清理。

head、tail和sweepVotes的更新都是使用cas进行更新的。

xfer的操作类型：

  1. NOW：包括的方法有非限时版本的poll和非限时版本的tryTransfer。这些操作不会阻塞等待，无论成功还是失败都会立即返回。
  2. ASYNC：包含异步插入的方法，包括add、put和offer，由于队列是无界队列，所以任何的插入方法都不会阻塞。
  3. SYNC：包含同步的方法，包括transfer()和take()方法。这两个方法都会阻塞。
  4. TIMED：包含timed的方法，包括poll(l,TimeUnit)和tryTransfer(l,TimeUnit)。

xfer()

xfer()方法尝试匹配已有节点。首先从head开始跳过所有已经匹配的节点，直到找到一个不同类型的尚未匹配的节点。如果找到了这样的节点，则匹配然后返回。如果队列中存在若干个已经匹配的节点，则更新head跳过所有已经匹配的节点，指向队列中第一个尚未匹配的节点(或者若果队列中没有未匹配节点，则使head指向当前节点)。

四个参数为：

  1. e：数据对象。当为请求类型时，为null。
  2. havaData：当为true表示节点为数据类型。当为false表示节点为请求类型。
  3. how：上面NOW、ASYNC、SYNC和TIMED四种操作类型之一。
  4. nanos：如果操作类型为TIMED，则参数为线程的等待超时时间。

这一部分属于匹配阶段：

  1. 从队列head开始一个个取已经插入的节点。
  2. 如果当前节点已经匹配成功，则取下一个节点。
  3. 否则当前节点未匹配，但是与参数中的节点统一类型，则退出循环。因为都是同一类型的节点，所以新节点需要插入队列。在后面代码中完成。
  4. 否则当前节点未匹配，且与参数中的新节点类型不同，则进行匹配。
  5. 匹配成功后，更新head，唤醒匹配节点对应的线程(如果线程阻塞的话)。
  6. 最后返回目标对象。

下面是参数节点与队列中的节点同类型时，需要插入队列的情况处理：

  1. 如果操作标识how为NOW，即poll()和tryTransfer()，这两个方法希望与队列匹配，但是匹配失败则立即返回。因为前面发现队列中的节点与参数节点是同类型的，所以匹配失败，即立即返回。
  2. 否则操作标识how不为NOW，新节点未创建则创建新节点对象。
  3. 调用tryAppend()方法插入队列尾部。
  4. 返回null时，发生多线程竞争插入失败，则重新进行一次xfer()操作。
  5. 否则插入成功，且操作是ASYNC，则不需要等待，直接返回目标对象。
  6. 否则插入成功，且操作不是ASYNC，则调用awaitMatch()进行等待。

tryAppend()

tryAppend()方法尝试在队列尾部插入新节点。如果新节点与tail所指向的节点同类型，或者tail指向的节点已经匹配成功，则插入尾部。否则如果head和tail都指向为null，则新节点作为队列的第一个节点。最后再进行更新tail的操作。

两个参数为：

  1. s：待插入队列尾部的节点。
  2. haveData：节点的类型。如果为true则表示是数据类型的节点，否则是请求类型的节点。

分为以下5种情况：

  1. 如果head和tail都为null，则说明需要初始化，就让head指向新节点，即新节点称为队列的第一个节点。
  2. 否则p.cannotPrecede()方法判断队列因为类型不同，不能插入则返回null。
  3. 否则如果p节点不是队列中最后的节点(发生多线程操作)，则p继续向后遍历。
  4. 否则将新节点进行cas插入队列的操作。
  5. tail需要更新。
    最后返回队列新节点在队列中的前驱节点。

awaitMatch()

当一个节点(数据或者请求类型的节点)插入到队列后，需要阻塞的情况就会调用awaitMatch()方法。对于一个等待的线程，等待的方式有三种：自旋、yield(cpu让步)和阻塞。当一个等待线程由于中断、超时或者被另外一个线程匹配成功时，这个等待线程会从等待中被唤醒。如果等待线程对应的节点是队列中的第一个未匹配的节点时，则采用front-queue的自旋方式，默认情况下最多自旋等待128次，在队列中未匹配的节点非常多时，这种front-queue的自旋等待方式对于性能的提高非常有效。而当队列中未匹配的节点较少时，这种front-queue的自旋等待方式对于性能的影响也不会很大。在等待线程自旋的过程中，根据线程本地变量(threadlocal-random)生成的随机数有一定几率使等待线程进入yield状态。这也可以提高性能，而且不会破坏spins和blocking的等待模式。然后对于前驱节点对应的线程当前处于自旋(而没有阻塞)等待状态的节点，采取chained-spins的自旋方式，一般是自旋次数是front-queue次数的一半。再者对于队列中的front线程，而且刚刚好队列的节点类型发生了变化(从数据类型变为请求类型或者相反，称为phase-changed)，则这个节点对应的线程自旋的次数为front-queue和chained-spins两者的和。最后如果在线程自旋完成后还没有完成匹配(中断或者超时)，则等待线程进入阻塞(LockSupport.park())。

总共5个参数：

  1. s：当前线程对应的节点。
  2. pred：当前线程对应节点的前驱。
  3. e：数据对象。
  4. timed：是否是timed方法。
  5. nanos：如果是timed方法，则nanos表示线程最大等待时间。
  1. 如果当前节点被其它节点匹配成功，则不需要自旋/yield/阻塞，将节点设置成已匹配的状态，然后直接返回。
  2. 如果当前线程被取消(中断、或者超时)，将节点置为已经取消，然后调用unsplice()清理当前节点，最后返回。
  3. 如果自旋次数没有确定，则调用spinsFor()方法确定自旋次数(三种里面的一种)，同时创建线程本地随机数生成的对象。
  4. 如果自旋次数确定，且自旋还没有到达预设值，则更新当前自旋次数，然后生成随机数并且决定是否进入yield。
  5. 如果自旋次数用完，则为当前节点设置waiter域，即当前线程。
  6. 如果是timed类型的方法，则当前线程阻塞并且随后更新仍然需要阻塞的时间。
  7. 否则不是timed类型的方法，则当前线程完全进入阻塞。

unsplice()

unsplice()方法除了可以减轻gc的回收压力外，还可以删除那些匹配成功或者取消的节点。这些节点出现的原因包括：中断、超时、被remove()和itor.reove()方法移除。一般在给定一个待移除节点s以及它的前驱pred的条件下，只要用cas更新pred的next域指向s的next域指向的对象就可以移除s。但是有两种特殊的情况不能保证s可以移除：1. 队列中的tail指向s，此时s的next域指向为null，由于队列的插入是尾插法，所以为了保证有节点插入成功，tail不能移除。2. 当s的前驱节点pred已经匹配或者取消，也调用了unsplice()进行清理时，此时pred的前驱的next域会指向s，而如果删除s，则会将pred的next域指向s的next域，这种s的删除方式没有用。所以对于上面两种情况会清理失败。为了解决这个问题使用一个sweepVotes变量记录因为遇到上面两种情况而出现的清理失败的次数，当失败次数达到SWEEP_THRESHOLD这个阀值时，则调用sweep()进行一次全局的清理。

两个参数：

  1. pred：待unlink节点s的前驱。
  2. s：待ublinke节点。
  1. 首先调用forgetContent()方法使item域指向自身节点，waiter域置为null，方便gc回收两个对象。
  2. 如果pred确实是s的前驱，则继续。
  3. 主要分为5种情况：
    3.1 s为tail且pred未匹配，则进入sweepVote清理阶段。
    3.2 s非tail且pred未匹配，则直接清理s，即更新pred的next域。
    3.3 s非tail且pred取消， 则进入sweepVote清理阶段。
    3.4 s非tail且pred匹配成功，不unlink节点s也不进入sweepVote。
    3.5 s为tail且pred取消或者匹配成功，则仅仅更新head，而不unlink节点s也不进入sweepVote

第四种和第五种的实现不同。
