Netty-服务器bind过程

echo代码

这里用的是netty.example中的echo代码进行分析

server端会涉及到四个handler:

  1. handler，对accpt线程对应channelHandler的初始化。在echo中就是一个LoggerHandler，直接插入即可。标记为1。
  2. childhandler，对每个连接到客户端的channel的handler的初始化，是一个ChannelInitializer。需要递归调用。标记为2。
  3. init方法中，匿名创建的ChannelInitializer。标记为3。
  4. 在3的initChannel()方法中添加的匿名ServerBootstrapAcceptor，是一个ChannelInitializer，标记为4。主要完成与客户端相连后获取的channel的参数配置和handler插入。

bind 初始化过程

  1. 从ServerBootStrap.bind()方法开始
  2. 进入initAndRegister()方法，首先新建一个NioServerChannel，然后调用init()方法初始化channel
  3. 还是在initAndRegister()方法中，调用config().group().register(channel)方法将channel注册到一个bossGroup中的eventloop中。注册过程中会将register()任务添加到选中的eventloop的任务队列中，并且创建新线程开始执行任务。
  4. 在主线程中，将bind()任务添加到选中的eventloop的任务队列中。
  5. 然后开始在被channel注册的eventloop线程中开始执行任务队列的任务。

任务队列的任务

首先确定任务队列中总共会出现4个需要执行的任务，当然这四个任务不可能同时出现在任务队列中：

  1. channel的register()任务
  2. 在init()方法中addLast()的ChannelInitializer会将一个runnable()任务添加到任务队列中。
  3. channel的bind()任务
  4. bind()任务执行完之后，在pipeline中传播Active事件的任务。

init()

init()方法目的是初始化channel，包含两个方面：

  1. channel的option和attr初始化。
  2. 向channel的pipeline对象，插入handler。

第一个过程很明显，主要看第二个过程。

首先addLast(handler)方法会将handler包装成一个ChannelHandlerContext，并且插入双链表中，然后决定是否立即执行addHandler事件的回调方法。对于回调方法，如果handler是一个ChannelInitializer，则执行inithandler()将handler添加到双链表，并且在双链表中删除这个ChannelInitialize。否则什么都不做。对于什么时候执行addHandler事件的回调方法，有三种情况：

  1. 如果pipeline对应的channel还没有注册到eventloop中，将ctx状态从INIT修改为ADD_PENDING的状态，并且调用callHandlerCallbackLater()，将这个addHandler事件触发的任务执行添加到到一个pipeline的pending任务链表中，当register()完成后调用。
  2. 如果pipeline对应的channel已经注册到eventloop，但是调用addLast()方法的线程不是eventloop线程，则将addHandler触发的任务添加到eventloop的任务队列中执行。
  3. 如果channel已经注册，而且调用addLast()代码的是eventloop线程，则直接指向addHandler触发的任务，即在双链表中添加handler。

注意:上面三种情况，主要是为了保证handler的初始化工作都是在channel对应的eventloop线程中执行，可以认为是netty非阻塞线程模型实现的一部分。

其实addLast()其实是一个循环调用的过程。

这里涉及到netty对channel的一个option设置，称为autoread(netty自动读配置)，
如果autoread = true，则当将channel绑定(bind)成功，或者与服务器连接(connnect)成功后，在传播active事件时，在head的handler中会调用channel.read()，将channel对selector设置的关注事件为readInterestOp，其中readInterestOp在构造时被指定，对于NioServerSocketChannel为OP_ACCEPT，对于NioSocketChannel为OP_READ。

channel.read()最后都会调用AbstractNioChannel.doBeginRead()方法：

配置channel的autoread为true的意思为：
这个channel可以接收数据，不管是服务器接收OP_ACCEPT，还是服务器或者客户端接收OP_READ事件。
如果配置channel的autoread为false的意思为：
这个channel不自动打开读数据的行为，默认打开。
